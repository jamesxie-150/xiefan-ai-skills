# Demo Project — 系统架构

## 整体架构

采用微服务架构，通过 API Gateway 统一入口，服务间通过 gRPC + 消息队列通信。

```
                    ┌─────────────┐
                    │  API Gateway │
                    │  (Kong)      │
                    └──────┬──────┘
            ┌──────────────┼──────────────┐
            ▼              ▼              ▼
    ┌──────────────┐ ┌──────────┐ ┌──────────────┐
    │ Order Service│ │ Payment  │ │  Logistics   │
    │  (Go)        │ │ Service  │ │  Service     │
    │              │ │ (Go)     │ │  (Python)    │
    └──────┬───────┘ └────┬─────┘ └──────┬───────┘
           │              │              │
           ▼              ▼              ▼
    ┌──────────────────────────────────────────┐
    │          PostgreSQL + Redis              │
    └──────────────────────────────────────────┘
```

## 服务清单

| 服务 | 语言 | 端口 | 职责 |
|------|------|------|------|
| api-gateway | — | 8080 | 路由、鉴权、限流 |
| order-service | Go | 8001 | 订单 CRUD、状态机 |
| payment-service | Go | 8002 | 支付渠道对接 |
| logistics-service | Python | 8003 | 物流追踪、路径规划 |
| notification-service | Python | 8004 | 消息推送 |
| web-console | TypeScript | 3000 | 管理后台前端 |

## 数据库设计要点

- **orders** 表：使用分布式 ID（snowflake），按月分表
- **payments** 表：支付记录与订单一对多
- **状态机**：订单使用有限状态机管理，禁止跳状态
